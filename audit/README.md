# Triage audit receipts

This directory holds **append-only audit receipts** written by `cli.py label`
on apply runs (not dry runs). Each receipt is the product's proof that the
**protected-sender gate held**: every message the engine touched is recorded with
its post-gate disposition, and every protected sender is recorded as held in the
inbox rather than archived.

The receipt is generated by `core/audit.py` (`AuditLog`), wired into the provider
chokepoint (`providers/base.apply_actions` and the Gmail `apply_actions` override)
alongside the gate itself.

## Why this exists

A safety guarantee you cannot *see* is one you have to take on faith. The audit log
is an **independent observer** of the gate, and the independence is real in two
specific ways:

1. **It records what the provider *actually did*** — whether `archive()` succeeded,
   whether `INBOX` was really removed, whether a folder move ran — not the
   (gate-mutated) action's intended fields. A regression where the gate fails to
   neutralize a protected message but the provider still carries the move out is
   therefore *observed as it happened*.
2. **It re-derives "is this sender protected?" itself**, by re-running the canonical
   `is_protected_sender` on the raw sender, rather than trusting the flag the gate
   passed in. So even a gate that stops *recognizing* a protected sender is caught.

If a sender the audit independently judges protected is observed leaving the inbox,
`AuditLog.assert_no_violations()` raises `AuditInvariantError` and the CLI exits
non-zero (code 2). The audit shares only the *definition* of "protected" with the
gate (which is correct, and unit-tested at the rules layer) — never the gate's
per-message decision.

## Format

One JSON object per line (JSONL). Example (non-protected, archived):

```json
{"ts":"2026-05-31T14:30:00+00:00","provider":"gmail","message_id":"18f...","disposition":"archived","protected":false,"archived":true,"moved":false,"labels_added":["Marketing"],"domain":"some-shop.example","dry_run":false,"sender":"Sale <promo@some-shop.example>"}
```

Example (protected, held — the trust signal):

```json
{"ts":"2026-05-31T14:30:00+00:00","provider":"gmail","message_id":"18a...","disposition":"protected_held","protected":true,"archived":false,"moved":false,"labels_added":["Finance/Banking"],"domain":"example-bank.com","dry_run":false,"sender":"Alerts <alerts@example-bank.com>"}
```

Dispositions: `protected_held`, `archived`, `moved`, `labeled`, `kept`.

## Privacy

Receipt lines contain **real sender addresses**. Two layers keep them out of git:
the repo-root rule `audit/*.jsonl`, and this directory's own `.gitignore`
(`*` except the docs), which also covers any sub-directory the root rule would miss.

Two caveats the CLI warns about at runtime:

- A **custom `--audit-file` path outside `audit/`** is NOT auto-ignored. If you
  redirect the receipt elsewhere, add it to `.gitignore` yourself or use
  `--redact-audit`.
- The default is **un-redacted** (full addresses). Run with `--redact-audit` to omit
  the local-part and store only the sender domain — a receipt you can share or attach
  to a support ticket.

If the receipt file cannot be written (disk full, unwritable path), the run does
**not** abort: the invariant is still checked in memory and the CLI prints a warning
that no file was persisted.
